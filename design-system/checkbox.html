Checkbox & Radio

Checkboxes let users select one or more items from a list. Radio buttons let users select exactly one option from a set. Both support label, supporting text, indeterminate state, and two sizes.

Stable WCAG AA Figma · 745-29926

Overview

Both controls share the same token set and behavior model — they differ only in selection logic and shape. Use them in forms, settings panels, filter sidebars, and any context where users need to choose from a predefined list.

Checkbox — multi-select
Radio — single-select

Types

Choose the type based on the selection logic — whether the user can pick one item or many.

Checkbox

4px border-radius. Multiple items can be selected simultaneously. Supports indeterminate state for "select all" patterns.

Radio

50% border-radius (circle). Exactly one item can be selected at a time. No indeterminate state.

Control Use when Avoid when
Checkbox Users can select zero, one, or multiple options. E.g. "Select features to enable", filter tags, permission toggles. Only one option is valid at a time. Never use checkboxes as radio button substitutes.
Radio Users must pick exactly one from a small set (2–6 items). E.g. billing cycle, gender, priority level. There are more than 6 options — use a Select instead. Never allow all radios to be deselected.

Do's & Don'ts

✓ Use checkboxes for multi-select

When users need to pick multiple items from a list, checkboxes clearly communicate that all selections are valid simultaneously.

✕ Don't use checkboxes as radio substitutes

If only one selection is valid at a time, use radio buttons. Checkboxes suggest multiple values are acceptable — misusing them creates false expectations.

✓ Use supporting text for context

Supporting text below the label gives users the additional context needed to make a confident decision without cluttering the label itself.

✕ Don't use overly long labels

Labels should be concise — ideally one line. Lengthy label text wraps awkwardly next to the small control and hurts scannability.

✓ Always provide a visible label

Every checkbox and radio must have an associated text label. Icon-only controls are inaccessible and confuse users who rely on screen readers.

✕ Don't mix sizes in the same group

All controls within a single group should use the same size. Mixing sm and md creates visual inconsistency and misaligned tap targets.

✓ Use indeterminate for partial selection

When a parent "Select all" checkbox has only some children checked, set it to indeterminate. This communicates the mixed state clearly and allows the user to check or uncheck all at once.

✕ Don't use radio buttons without a default

Always pre-select a sensible default radio option. An empty radio group forces users to make a choice before they can proceed, which increases cognitive load and error rates.

Accessibility

Checkbox and Radio are keyboard navigable, screen-reader compatible, and meet WCAG 2.1 AA contrast requirements in all states.

ARIA Roles & Attributes

  • Native input[type=checkbox] and input[type=radio] provide correct roles automatically
  • Use aria-checked="mixed" for indeterminate checkbox state
  • Label must be associated via htmlFor or wrapping <label>
  • Group radios in <fieldset> with <legend> to convey group context

Keyboard Interaction

  • Tab — moves focus to the control
  • Space — toggles checkbox; selects focused radio
  • Arrow keys — navigate between radio options in a group
  • Focus ring uses box-shadow so it renders inside overflow:hidden containers

Color & Contrast

  • Unchecked border (#bdbdc8 on white) meets WCAG AA 3:1 non-text contrast
  • Checked fill (#7f56d9) against white checkmark passes WCAG AA 4.5:1
  • Label text (#29294c) on white background: contrast ratio ≥ 7:1
  • Supporting text (#707087) on white: meets 4.5:1 for small text

Touch & Target Size

  • Minimum touch target is 44×44px (the wrapping <label> expands the hit area)
  • Include label text to maximize tappable area
  • Maintain at least 8px vertical gap between items in a group
  • Never rely on hover state alone — all state information is also available at touch level

Indeterminate state (JavaScript)

// Set indeterminate state on the native input
const cb = document.querySelector('#select-all');
cb.indeterminate = true;

// Sync visual control class
function updateParent(parentControl, children) {
  const checked = [...children].filter(c => c.checked).length;
  const total   = children.length;
  if (checked === 0) {
    parentControl.classList.remove('is-checked', 'is-indeterminate');
  } else if (checked === total) {
    parentControl.classList.add('is-checked');
    parentControl.classList.remove('is-indeterminate');
  } else {
    parentControl.classList.add('is-indeterminate');
    parentControl.classList.remove('is-checked');
  }
}

Implementation Notes

For Designers

  • Figma component node: 745-29926
  • Use the Type prop to switch between Checkbox and Radio
  • The Indeterminate prop only appears when Type = Checkbox
  • Supporting text is optional — detach and remove if not needed
  • Maintain 8px vertical gap between items in all groups
  • Never modify the control shape or color tokens directly — override via the design token layer
  • Use the auto-layout frame to build groups — avoid manual positioning

For Developers

  • Always use native <input> — hide visually, not from the accessibility tree
  • Set input.indeterminate = true in JS (not an HTML attribute)
  • Keep radio buttons grouped with the same name attribute
  • Wrap groups in <fieldset>/<legend> for screen reader context
  • The .is-checked class on .cb-control drives the visual state — keep it in sync with the native input
  • Disabled state: add .is-disabled to the wrapper AND disabled to the native input
  • In React, use controlled components and call onChange to update state
